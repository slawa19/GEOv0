# Канонические правила документации (RU)

Этот документ фиксирует **единые правила** для русскоязычной документации проекта GEO.

## 1. Зачем нужны правила

Цели:

- отделить **общую документацию** от **доменных спецификаций** и от **предварительных концепций/статей**;
- обеспечить предсказуемую навигацию и стабильные точки входа;
- минимизировать дублирование и устаревшие ссылки.

## 2. Структура `docs/ru/*` (канон)

### 2.1. Общая документация (stable)

Каркас общего описания системы — файлы верхнего уровня:

- `docs/ru/00-overview.md`
- `docs/ru/01-concepts.md`
- `docs/ru/02-protocol-spec.md`
- `docs/ru/03-architecture.md`
- `docs/ru/04-api-reference.md`
- `docs/ru/05-deployment.md`
- `docs/ru/06-contributing.md`
- `docs/ru/08-test-scenarios.md`
- `docs/ru/09-decisions-and-defaults.md`
- `docs/ru/10-testing-framework.md`

**Правило:** эти документы — «ворота» в проект. Они должны ссылаться на доменные README и не содержать подробных UI-спек на десятки страниц.

### 2.2. Доменные области (stable)

Доменные области содержат спецификации и руководства по конкретным подсистемам.

- `docs/ru/admin-ui/*` — документация админского UI (и связанной операторской логики)
- `docs/ru/backend/*` — документация backend и API реализации hub
- `docs/ru/simulator/*` — документация приложения-симулятора поведения

**Правило:** каждая доменная область имеет входную точку `README.md` и внутри собственную навигацию.

### 2.3. Предварительные концепции и статьи (draft)

Все материалы, которые являются:

- черновыми идеями,
- исследовательскими заметками,
- диалогами,
- статьями,

живут в `docs/ru/concept/*`.

**Правило:** `concept` — не источник истины для требований к реализации. Если документ в `concept` стабилизировался, его нужно перенести в `docs/ru/*` (общие доки) или в домен.

### 2.4. Архив (archived)

Устаревшие, заменённые или исторические документы — `docs/ru/archive/*`.

**Правило:** архив не правим содержательно (кроме минимальных пометок о статусе/замене), чтобы не искажать историю.

## 3. Статусы документов

Допустимые статусы:

- **Stable** — актуальный документ, является источником истины.
- **Draft** — рабочий материал, допускает противоречия и неполноту.
- **Archived** — исторический документ, не является источником истины.

Рекомендуемый блок метаданных в начале (опционально для старых файлов):

```text
Статус: Stable | Draft | Archived
Область: core | backend | admin-ui | simulator | concept
Последнее обновление: YYYY-MM-DD
```

## 4. Нейминг и переносимость

### 4.1. Имена файлов

- Для **новых** документов в доменах предпочтителен ASCII `kebab-case`.
- Исторические RU-файлы с пробелами/длинными названиями не переименовываем без необходимости; при переносе допускается оставить имя как есть.

### 4.2. Ссылки

- Внутри `docs/ru/*` использовать **относительные** ссылки.
- Ссылки должны указывать на актуальные пути после переносов; «старые пути» не поддерживаются (stub-файлы не оставляем).

## 5. Разделение ответственности

- Доменные документы не должны дублировать общий каркас; вместо этого — ссылка на раздел общего каркаса.
- Общий каркас не должен «поглощать» детальные спецификации доменов; вместо этого — ссылки на доменные README.
