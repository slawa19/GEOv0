# WM Post-Refactor UX Issues — spec (2026-03-01)

## Контекст
После рефакторинга окон (WindowManager + WindowShell) остались реальные UX-проблемы, которые **не считаются решёнными тестами**, т.к. проявляются в живом UI.

Связанные документы:
- `docs/ru/simulator/frontend/docs/specs/window-manager-acceptance-criteria.md` — базовые критерии/карта реализации

Цель этого документа: **зафиксировать симптомы, воспроизведение, ожидаемое поведение, гипотезы причин и варианты решений**. Реализация должна быть начата **только после ревью этого документа**.

## Scope / Non-goals
В scope:
- WM-UX-1: «Родительское Action окно сплющено» (визуальные размеры/паддинги/лейаут)
- WM-UX-2: «Родительское окно закрывается при открытии Send Payment»

Non-goals (для этой итерации):
- Редизайн компонентов или изменение дизайн-системы
- Добавление новых окон/страниц/сложных анимаций
- Любые изменения бизнес-логики платежей/клиринга

## Определения
- «Родительское Action окно» — окно-инспектор, из которого запускается действие (например, Edge Detail / Node Card в режиме WM).
- «Дочернее окно» — Interact-панель (Manual Payment / Trustline / Clearing), которая открывается для выполнения действия.

## WM-UX-1 — Action окно «сплющено»

### Симптом
Родительское Action окно в WM режиме выглядит «сплющенным»: нарушены ожидаемые размеры, внутренние отступы, сетка элементов, заголовок/контент не выглядят как остальные WM окна.

### Влияние
- Визуальная деградация после рефакторинга
- Пользовательская уверенность падает (ощущение «сломано»)
- Косвенно влияет на кликабельность/читаемость кнопок

### Минимальные шаги воспроизведения (manual)
1) Открыть Simulator UI v2: `/?mode=real&ui=interact` (WM по умолчанию включён)
2) Открыть родительское Action окно (Edge Detail или Node Card)
3) Сравнить визуально с другими окнами (WindowShell), особенно по:
   - высоте/ширине и минимальным размерам
   - паддингам внутри тела
   - поведению overflow (скролл vs сжатие)

### Ожидаемое поведение
- Родительское окно соответствует единому контракту `WindowShell`:
  - адекватные minWidth/minHeight (не меньше “читабельного” состояния)
  - внутренний контент не сжимается за счёт flex/grid ошибок
  - если контент не помещается — появляется корректный scroll внутри body, а не «сдавливание»

### Текущее поведение (наблюдаемое)
- Окно визуально меньше ожидаемого; контент выглядит сжатым.

### Гипотезы причин (не утверждения)
H1. CSS контента внутри `WindowShell` использует `height: 100%` + flex/grid так, что элементы получают `min-height: auto` и сжимаются.
H2. Размерные constraints в WM для данного типа окна слишком агрессивные (малые preferred/min), и measured size не обновляет rect ожидаемым образом.
H3. Внутренний компонент (EdgeDetail/NodeCard) оставляет legacy-стили overlay-панелей, которые конфликтуют с WM контейнером.
H4. `WindowShell` body или content wrapper имеет `overflow: hidden`/`min-height: 0`/`align-items` настройки, провоцирующие компрессию.

### Варианты решений

**Option A — Нормализовать layout контента через контракт “WM-content-only”**
- Для компонента родительского окна явно определить WM-режим рендера (аналогично Interact панелям): без legacy overlay оболочки, с корректным root layout.
- Привести root контейнер к совместимости с `WindowShell`:
  - не задавать конфликтующие width/height/maxWidth
  - обеспечить, что скролл управляется `WindowShell` body (или явным inner wrapper)

Плюсы:
- Стабилизирует визуально и упрощает контракт
- Убирает скрытые зависимости от legacy overlay CSS

Риски:
- Можно задеть существующие “legacy” режимы (fixtures/offline) — нужна проверка обоих режимов

**Option B — Исправить `WindowShell` layout-контракт (body/content)**
- Правка в одном месте: обеспечить корректное поведение flex/grid для любых детей
- Явно определить:
  - `min-height: 0` на нужных flex контейнерах
  - overflow policy (scroll) на body

Плюсы:
- Системное исправление для всех окон

Риски:
- Можно неожиданно изменить поведение других окон (регрессии)
- Потребует более широкого визуального ревью

**Option C — Тюнинг constraints в WM (preferred/min размеры)**
- Увеличить preferred/min размеры конкретного window type

Плюсы:
- Быстро и точечно

Риски:
- Это может “маскировать” проблему layout’а и не решить паддинги/overflow
- На маленьких viewport может ухудшить clamping и UX

### Acceptance Criteria (WM-UX-1)
AC1. Родительское Action окно в WM имеет консистентные паддинги/типографику и не выглядит «сплющенным».
AC2. На узком viewport контент не сжимается “в кашу”: появляется предсказуемый scroll внутри окна.
AC3. В legacy режиме (c `wm=0`) визуальное поведение не ухудшается.

## WM-UX-2 — Родительское окно закрывается при Send Payment

### Симптом
При нажатии “Send Payment” из родительского Action окна, родительское окно **закрывается**, хотя ожидается, что оно останется открытым как контекст.

### Влияние
- Нарушение контракта “контекст остаётся, действие поверх”
- Потеря ориентации пользователя

### Минимальные шаги воспроизведения (manual)
1) Открыть `/?mode=real&ui=interact` (WM по умолчанию включён)
2) Открыть Edge Detail (инспектор ребра)
3) Нажать “Send Payment”

### Ожидаемое поведение
- Открывается Manual Payment как отдельное окно в группе `interact`
- Edge Detail остаётся открытым (группа `inspector`), не закрывается programmatic-логикой

### Текущее поведение (наблюдаемое)
- Edge Detail закрывается при запуске Send Payment.

### Гипотезы причин (не утверждения)
H1. Обработчик “Send Payment” вызывает `cancel()`/очистку interact state, и watcher, который синхронизирует окно инспектора с interact state, закрывает его programmatic.
H2. В разных UI-путях используются разные обработчики, и keep-alive логика не активируется во всех местах.
H3. В `WindowShell` close/replace политика или “cross-group replace” логика по ошибке закрывает инспектора.

### Варианты решений

**Option A — Декуплинг: inspector window живёт независимо от interact state**
- В WM-режиме инспектор определяется выбором/окном, а не transient state FSM.
- “Send Payment” не должен обнулять данные, на основании которых инспектор остаётся открыт.

Плюсы:
- Самая стабильная модель: WM — единственный источник правды для открытости окна

Риски:
- Потребуется аккуратная интеграция “selection changes” → update inspector

**Option B — Явная keep-alive семантика “parent окно остаётся”**
- В момент Send Payment выставлять флаг keepAlive и (опционально) freeze данных для инспектора
- Watcher закрытия инспектора должен учитывать keepAlive и **не закрывать** окно
- Обязательно обеспечить, что это работает для всех UI путей (кнопка внутри EdgeDetail, хоткеи, любые action-бар кнопки)

Плюсы:
- Минимально инвазивно

Риски:
- Легко пропустить один из путей; будет “иногда работает”

**Option C — Parent/child relation в WM (стэк/связь окон)**
- При открытии payment окна сохранять ссылку на parent window id
- WM гарантирует, что parent не закрывается при open child

Плюсы:
- Формализует контракт

Риски:
- Усложнение WM модели (MVP может быть избыточно)

### Acceptance Criteria (WM-UX-2)
AC1. В WM (по умолчанию) Edge Detail остаётся открыт при “Send Payment”, при этом Manual Payment открывается поверх.
AC2. ESC-policy работает по правилам: сначала закрывается interact окно, затем inspector.
AC3. В legacy режиме поведение не ухудшается.
