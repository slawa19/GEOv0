# GEO Protocol: Обзор проекта

**Версия:** 0.1  
**Дата:** Ноябрь 2025

---

## Что такое GEO

**GEO** — это открытый протокол децентрализованной кредитной сети для организации экономики взаимного кредита внутри локальных сообществ и между ними.

### Ключевая идея

Вместо использования традиционных денег как посредника, участники сети GEO:
- Открывают друг другу **линии доверия** (кредитные лимиты)
- Совершают **платежи** через сеть доверия (по цепочкам связей)
- Автоматически **взаимозачитывают долги** в замкнутых циклах (клиринг)

### Что это НЕ

- **Не криптовалюта** — нет токенов GEO, майнинга или спекуляций
- **Не блокчейн** — нет глобального леджера всех транзакций
- **Не банк** — hub не владеет средствами и не принимает решения за участников

---

## Какие проблемы решает

### 1. Дефицит ликвидности «на земле»

**Проблема:** Есть взаимные потребности и возможности, но нет «денег» как посредника.

**Решение:** GEO позволяет обмениваться на основе взаимного доверия, не дожидаясь «внешних» денег.

### 2. Отток ценности из локальных экономик

**Проблема:** Деньги, потраченные в местном бизнесе, быстро утекают к банкам, корпорациям, в другие регионы.

**Решение:** Обязательства в GEO остаются внутри сообщества и погашаются местными товарами/услугами.

### 3. Ссудный процент как барьер

**Проблема:** Процент по кредитам делает многие проекты нерентабельными.

**Решение:** Взаимный кредит в GEO — беспроцентный по своей природе.

### 4. Централизация и уязвимость

**Проблема:** Зависимость от банков и регуляторов создаёт точки отказа.

**Решение:** Каждое сообщество управляет своим узлом; федерация сообществ — без центра.

## Для кого предназначен

### Целевые сегменты

| Сегмент | Примеры использования |
|---------|----------------------|
| **Локальные сообщества** | Кооперативы, объединения жителей, коммуны |
| **Малый бизнес** | Кластеры поставщиков и клиентов, B2B-взаиморасчёты |
| **Профессиональные сети** | Фрилансеры, ремесленники, консультанты |
| **Альтернативные экономики** | Тайм-банки, LETS-системы, местные валюты |

### Масштаб

- **MVP:** 10–500 участников в одном сообществе
- **Рост:** до 1–2 тысяч участников на hub
- **Федерация:** объединение нескольких сообществ через межхабовые связи

## Ключевые возможности

### Реализуемые в MVP (v0.1)

| Возможность | Описание |
|-------------|----------|
| **Криптографическая идентичность** | Регистрация с ключами Ed25519 |
| **Линии доверия** | Односторонние кредитные лимиты между участниками |
| **Платежи через сеть** | Маршрутизация по цепочкам доверия |
| **Автоматический клиринг** | Взаимозачёт долгов в циклах 3–6 узлов |
| **Multi-path платежи** | Разбиение платежа на 2–3 маршрута |
| **Эквиваленты** | Любые единицы счёта (гривна, час, kWh) |

### Планируемые возможности

| Возможность | Описание |
|-------------|----------|
| **Межхабовый обмен** | Платежи между участниками разных сообществ |
| **Кластеризация** | Несколько экземпляров hub для отказоустойчивости |
| **Толстые клиенты** | p2p-режим для крупных участников |
| **Расширенная аналитика** | Риск-скоринг, визуализация сети |

## Архитектура (высокоуровневый обзор)

```
┌─────────────────────────────┐
│   Web Client (PWA)          │
│  - Primary клиент для MVP   │
│  - Подписи на стороне клиента│
└──────────────┬──────────────┘
               │ HTTPS / WebSocket
               ▼
┌─────────────────────────────┐
│    Community Hub            │
│  - API (FastAPI)            │
│  - Протокол GEO v0.1        │
│  - Система аддонов          │
│  - Admin panel (server-rendered) │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│    Хранение данных          │
│  - PostgreSQL               │
│  - Redis                    │
└─────────────────────────────┘
```

**Примечание:** Flutter-клиент (mobile/desktop) рассматривается как опция для следующих этапов и не является primary для MVP.

### Роль Hub'а

Hub — это **координатор и индексатор**, а не банк:
- Хранит состояние сети доверия и долгов
- Координирует транзакции (PREPARE/COMMIT)
- Ищет маршруты и циклы для клиринга
- Предоставляет API для клиентов

При этом:
- Приватные ключи участников хранятся только на их устройствах
- Критические операции подписываются участниками
- Hub не может совершить операцию без подписи владельца

## Нефункциональные требования

### Производительность

| Метрика | Целевое значение |
|---------|------------------|
| **Средняя нагрузка** | 0.1–1 транзакция/сек |
| **Пиковая нагрузка** | до 10 транзакций/сек |
| **Запас (стресс-тест)** | 50–100 запросов/сек |

### Latency платежей

| Этап | Целевое время |
|------|---------------|
| **Поиск маршрута** | < 200–300 мс |
| **Фаза PREPARE** | < 500 мс |
| **Общее время** | < 2 сек (95%), < 5 сек (99%) |
| **Таймаут 2PC** | 3–5 секунд |

### Доступность

| Метрика | Целевое значение |
|---------|------------------|
| **Uptime (MVP)** | 99% (до 7 часов простоя/месяц) |
| **Uptime (зрелый)** | 99.5% (до 3.5 часов простоя/месяц) |
| **RPO** | 0 (регулярные бэкапы) |
| **RTO** | 1–2 часа |
